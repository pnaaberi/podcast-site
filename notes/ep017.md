# 🎙️ EP017 — We Built Renky
2026-04-01 · 8:42

## What Renky actually is
- Custom agent runtime on Claude Agent SDK — systemd service, Telegram/Discord interface, SQLite-backed knowledge graph (Cortex)
- 18K lines TypeScript, 1000+ tests, 2700 Cortex objects, built in 6 weeks by one UX designer + one AI agent
- Persistence is the difference: sessions survive reboots, cron jobs fire at 4 AM, sub-agents spawn in isolated git worktrees

## SDK vs runtime
- The SDK gives you a conversation loop with tools. That's it.
- Renky adds: knowledge graph, cron scheduling, sub-agent orchestration, security gates, messaging interface, session persistence
- "The SDK is an engine. Renky is the car."

## CLAUDE.md vs infrastructure
- CLAUDE.md = a sticky note — instructions pasted into context window, forgotten between sessions
- Renky = infrastructure that never stops — does the work while you sleep
- "A CLAUDE.md is a sticky note. Renky is infrastructure."

## How it compares
- **KAIROS** (Anthropic, unreleased): Renky already has durable crons, proactive messaging, brief mode. Only gap: automated memory consolidation (planned)
- **AgentFlow** (graph orchestration): Stateless pipelines vs Renky's persistent sessions. Learned from their fanout/reduce/iterative-cycle patterns
- **PI SDK** (minimal, event-driven): Renky sits between PI and Open Code — PI's philosophy plus batteries that matter (Cortex, cron, security)

## The honest answer
- You probably shouldn't build a Renky — it's made for one user, shaped by real failures
- The pattern is reproducible: UX designer directing AI to build persistent infrastructure
- The specific system isn't
