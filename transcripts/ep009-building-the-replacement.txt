[JENNY]
The Security Officer. Tell Chris about the "lock on the outside."

[JAKE]
Every framework puts security in the system prompt. "Ask before doing dangerous things." Lock on the inside of the door. Agent sees it, reasons about it, context overflow can evict it.

We use a callback that fires before every tool execution. Agent never sees it. Can't manipulate it. Tool calls get classified, checked against an allowlist, approved or denied before the agent knows it tried.

[CHRIS]
I like the concept. But the classification — you're parsing bash commands with regex?

[JAKE]
Yes. And I know. Bash parsing with regex is fundamentally limited. Base64 encoding, variable expansion, pipe chains. We documented six bypass scenarios. Four pass, two partial. The design doc says "bash parsing is best-effort, not a security boundary."

[CHRIS]
So the lock on the outside has holes.

[JAKE]
A lock on the outside with holes beats a lock on the inside the agent can just decide to ignore. Tier one catches obvious violations. Tier three escalates to the human with auth codes. The gaps are documented. We don't pretend it's airtight.
