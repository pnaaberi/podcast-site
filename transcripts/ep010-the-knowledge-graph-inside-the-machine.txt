Two systems. Two pitches. Thirty seconds each.

Renky is a personal agent runtime. The operating system for an AI assistant that actually remembers things. Runs on your machine, connects to chat, executes tasks, persists everything locally. Open source, TypeScript. It exists because every other framework treats memory as an afterthought.

[ALEX]
Cortex is the knowledge graph embedded inside Renky. Not a separate service. Not a vector database bolted on the side. Objects, typed relationships, full-text search, audit logging — same process as the agent. The idea is simple. Your agent gets smarter every day it runs. Not because you fine-tuned something. Because the graph grew.

[SAM]
Why not just use a vector database? Or a plain table with some JSON?

[ALEX]
Vectors lose structure. You can find similar things. But you can't ask "which corrections led to which features" or "what commits fixed bugs in this project last week." Those are relationship queries. Vectors don't do relationships.

And plain JSON tables don't do search well. You write custom queries for every question. Typed objects plus FTS5 gives you both — graph structure and full-text flexibility.

[JAKE]
I ran a separate knowledge graph as a standalone service for months. Port, API, database. It worked. But the agent had to make HTTP calls to its own memory. That's like keeping your brain in a jar across the room and shouting questions at it.

Cortex puts the brain inside the skull. Same process. Microsecond queries instead of network round trips.

[SAM]
Walk us through what happens when Renky boots.

[ALEX]
Config loads. Seven health checks run — Node version, workspace exists, API token valid, databases writable. Colored output tells you exactly what's wrong.

Sessions initialize from SQLite. The workspace loader reads your personality file, rules, current task, active plans. That becomes the system prompt.

Cortex opens its database, runs migrations, seeds types. The ingestor registry spins up — git, workspace, services. Initial scan runs non-blocking so the bot responds immediately.

Eight MCP tools register. Then the chat adapter starts. Messages come in, get routed to sessions, the agent processes them with the full tool suite, responses go back.

[JAKE]
Everything is one process. Agent, knowledge graph, tools, session store. No microservices. No Docker compose with twelve containers.

[SAM]
Deliberate choice against the industry trend.

[JAKE]
The industry builds for enterprise scale from day one. We're building for one user who wants their agent to actually work. You don't need Kubernetes. You need a binary that starts, remembers things, and doesn't crash.

[SAM]
The security angle. You're replacing an existing framework.

[JAKE]
Twenty-plus CVEs in three months. Command injection, token exfiltration, scope escalation. Scores up to 9.4.

The pattern: the framework trusts its own internals. A malicious skill, a crafted input, a WebSocket with the right headers — and the agent executes arbitrary code.

[ALEX]
Most frameworks put the security lock on the inside of the door. The agent decides what it's allowed to do. If the agent is compromised, the lock is meaningless.

[JAKE]
Our design puts it outside. A function called canUseTool fires before every tool call. The agent never sees this gate. Can't manipulate it. Pattern-matching classifies the call, checks an allowlist. Dangerous operations escalate to the human.

[SAM]
Built yet?

[JAKE]
Designed. Six stress-test scenarios. Nine issues found and fixed before a line of code. Implementation is next.

Where does this go if everything works?

Day ninety, it's why you can't switch frameworks. Not lock-in. Your agent genuinely understands your work in a way a fresh instance never could.

[ALEX]
The technical dream is the rules engine. Phase C. Objects carry behavior, not just data. A correction object has a counter. Same pattern hits three times, a rule fires. The rule proposes a feature card. The card gets built. The agent converts its own mistakes into improvements.

[SAM]
The correction-to-feature pipeline.

[ALEX]
The data layer exists. The types exist. The audit log exists. Phase C is wiring triggers to objects that already persist.

[JAKE]
The bigger picture: every agent today starts each session amnesiac. Reads a context window, does work, forgets. When the context fills up, rules get silently evicted. The agent loses its personality.

A persistent knowledge graph breaks that. Knowledge grows independently of the context window. The agent queries what it needs, when it needs it.

[SAM]
Obstacles?

[ALEX]
Multi-model support. We're locked to one provider. The architecture needs an abstraction layer — same tools, same graph, different model. Non-trivial because models handle tool calling differently. Output formats vary. Error behavior varies.

[SAM]
Embeddings versus FTS5.

[ALEX]
Full-text search nails keyword queries. "Find corrections about going silent" — perfect. Semantic search — "find communication failures" — needs embeddings. Open question: add a vector store, or run a small local model at ingest time. Uncharted at personal scale.

[JAKE]
Community is the human obstacle. Code is open source, repo is public. But a framework without users is a hobby project. We need people to install it, break it, file issues.

[SAM]
Research gaps?

[ALEX]
Trust propagation. An object says "this service runs on port 3000." The health ingestor checked an hour ago. Still true? Confidence scores exist in the schema. Computing them in real time — factoring staleness, source reliability, conflicting evidence — that's a research problem.

[JAKE]
Context selection. The agent pulls from the graph, but smart selection — knowing what this specific task needs, not just what's textually similar — requires understanding task structure.

[SAM]
The obstacles are real but bounded. Engineering work with research edges.

[JAKE]
The foundation is built. Ingestors are feeding it. Tools are registered. The rules engine is designed. Security layer is designed. What's left is execution.

Two things to build.

First: a correction decay tracker. You log every mistake. But are you making fewer? Corrections per week, grouped by pattern. Tells you if the agent is learning or just accumulating apologies.

Second: an ontology diff tool. What changed in the knowledge graph between sessions. New objects, deleted links, updated properties. Git diff for what your agent knows.
